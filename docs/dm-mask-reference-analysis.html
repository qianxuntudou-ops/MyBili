整理版：把 Web 实测、B 站 APP 端 smali、Chronos native 反汇编、MyBLBL 当前实现状态合到一条主线里。旧文档中的历史判断已收束，避免继续把已废弃结论当成当前事实。

1. 最终结论

推荐基准：数据按 Web，行为按 Android APP，渲染语义按 Chronos native。不要只照 Web，也不要硬抄 APP 主进程。

Web 端暴露了最容易验证的 x/player/v2.dm_mask 和真实 .webmask 文件；Android APP 端更接近 MyBLBL 的运行环境，播放器状态、弹幕生命周期、时钟同步都应该以它为主。Chronos native 则给出最关键的渲染语义：弹幕内容先完成绘制，再乘 mask alpha。

模块优先参考理由
mask 元信息Web / Android 都可字段本质一致：cid / plat / fps / time / maskUrl。MyBLBL 当前走 Web API 更实际。
.webmask 文件格式Web 实测能直接下载验证，已确认真实段结构和帧 PTS。
播放器 clockAndroid APP / Chronos参考通过 onPlayerClockChanged(rate, position, timestamp) 校准内部时钟，再按 rate 连续推进。
seek / buffering / readyAndroid APP这些是 Android 播放器状态语义，Web 环境差异太大。
渲染合成Chronos native反汇编证实 shader 读取 cron_mask_texture alpha，再乘弹幕输出。
最终实现MyBLBL 工程折中我们没有 Chronos native surface，只能用 View tree 中的 saveLayer + DST_IN 保留可显示 path。

段内帧不是纯字符串拼接，而是二进制帧流：

uint32 payloadLength   // big-endian，payload 字节数
uint64 presentationMs  // big-endian，视频 PTS，单位 ms
byte[payloadLength] payload

payload 才以 data:image/svg+xml;base64, 开头。第 0 段开头实测：

frame 0: len=1318 pts=0   payload startsWith data:image/svg+xml;base64

frame 2: len=518  pts=67  payload startsWith data:image/svg+xml;base64
工程结论：fps=29 只能作为元信息，不能作为段内帧时间的唯一依据。选帧必须优先使用每帧头里的 presentationMs。旧的字符串切分会把下一帧 12B 帧头混入上一帧 base64 尾部，造成解码失败和伪空帧。

3. Android APP 端参考链路

APP 端普通视频的主体防挡入口不是 Web JSON，而是 DmViewReply.mask 中的 VideoMask。字段结构和 Web 端 dm_mask 对齐：

字段类型含义工程影响
cid_long视频 CID。用于绑定当前视频，避免跨视频复用。
plat_int平台/CDN 选路标识。MyBLBL 当前不主动使用，但字段要保留。
fps_intmask 帧率元信息。不能替代帧头 PTS。
time_longmtime 或服务端生成时间。多用于 HTTP 缓存协商，不是渲染补偿。
maskUrl_Stringwebmask URL。空串要按不可用处理。

3.1 主 APK 是 RPC 客户端

smali 扫描后的关键判断：

  • 主 APK 会拿到完整 DmViewReply，并通过 Chronos RPC 附件传给远端。
  • DmViewReply.hasMask() 在主 APK 中主要用于 dm_mask 三态上报。
  • 全 smali 中没有发现主 APK 自己调用 VideoMask.getMaskUrl() 下载和渲染 webmask 的路径。
  • 播放器 clock 通过 RemoteServiceHandler.onPlayerClockChanged(rate, position) 发送给 Chronos。

因此，APP 主进程不是防挡渲染的真正消费者。它更像 RPC stub：收集 DmViewReply、开关和播放器状态，然后交给 Chronos native 进程。

3.2 播放器 clock 语义

参考链路不是每帧直接读取静态 player.currentPosition。它会把 current_time、playback_rate、timestamp 发给 Chronos，Chronos 内部再按系统时间和 rate 连续推进。

current_time + max(0, systemTimeMs() - timestamp) * playback_rate

MyBLBL 的等价策略应该是：用播放器 position 校准基准点，再用 elapsedRealtime * playbackSpeed 外推当前 PTS。不要把 video frame metadata PTS 作为第二套时间源。

4.2 定点反汇编结论

通过 .eh_frame 恢复到 24,124 个函数边界，再从关键字符串反查 xref，得到下面几个结论。本节按证据强度排列，避免把弱线索误当成主链路。

证据反汇编结论对 MyBLBL 的意义
cron_mask_texture / cron_mask_tex_coord 被 GPU/shader 相关函数引用，并和 texture/uniform 管线相邻。 mask 是纹理输入，不是简单 Canvas 裁剪。
mask_sample = texture(...).a shader 读取 mask texture 的 alpha。 防挡核心是 alpha mask。
CRONDefaultShading() * mask_sample 弹幕默认输出乘 mask alpha。 saveLayer + DST_IN 是 View 体系下最接近的近似：先画弹幕，再保留 webmask 可显示区。
android/graphics/Path / setClipPath Chronos 有 Android Path 和 clip/path 节点能力。 这只能证明 Chronos 支持 Path，不能推翻 shader 证据。
bad zlib header / zlib corrupt 附近函数是内置 deflate/zlib 解压逻辑。 Chronos 具备自己解压资源的能力，但还不能直接还原完整 webmask 消费状态机。
MASK 被一个大型解析函数引用，旁边有大量字符串比较。 不能单独证明这里就是 B 站 .webmask 解析器，只能作为弱证据。

6. 已知问题分层

后续排查不要把所有闪烁都归到智能防挡。当前至少有三类问题，要分开看。

现象优先归因判断方式下一步
人物区域整块“遮住又露出来” webmask 解析失败伪空帧，或 Host 状态切到无遮罩。 看 segment 是否已缓存、frame 是否为 null、是否出现 decode failed。 二进制帧解析已修正后，继续看缺段和 shouldRenderMask=false 日志。

7. 参考逻辑整理

  1. 视频详情或播放器接口返回 mask 元信息。Web 是 x/player/v2.dm_mask，APP 是 DmViewReply.mask / VideoMask。
  2. 客户端拿到 maskUrl 和 fps。fps 是元信息，段内帧时间以帧头 presentationMs 为准。
  3. 真实 .webmask 文件由 MASK 文件头、段索引表、gzip 段组成。段内每帧是 uint32 len + uint64 pts + payload。
  4. 播放器通过 rate + position + timestamp 校准内部 clock。后续按 elapsed time 和 playback rate 连续推进。
  5. 按当前 clock PTS 定位 segment 和 frame，使用 floor 语义，不取未来帧。
  6. 把 SVG path 转成弹幕可显示区。实机验证 DST_OUT 会反向显示，因此解析后的 path 应按可显示 alpha 处理。
  7. 弹幕内容先正常绘制，再按 mask alpha 保留可显示区。Chronos native 是 shader；MyBLBL 用 saveLayer + DST_IN 近似。
  8. BUFFERING/READY 是 clock/status 校准，不等价于清空弹幕 render queue。

8. 剩余差异和建议

优先级差异影响建议

10. 结论回收

旧文档里一些历史判断已经不再作为当前结论使用：

  • “webmask 是纯 base64 字符串列表”已废弃。真实结构是二进制帧流。
  • “人物区域整块露出是资源空帧”已废弃。样本 37,090 帧都有 path。
  • “主 APK 与 Chronos 共用 clock”表述过度简化。准确说法是主 APK 把 clock 转发给 Chronos。
  • “父 View clipPath 等价官方”已废弃。Chronos 证据指向 shader alpha 乘法。
  • “webmask path 必须用 DST_OUT 挖掉”已废弃。实机验证 DST_OUT 会让人物区域显示弹幕、其他区域无弹幕；当前应使用 DST_IN 保留解析后的可显示区。
  • “seek 后冻结旧 mask 防闪”不是参考主线。它能掩盖空窗，但会制造旧 PTS 遮罩拖尾。
  • “关闭智能防挡仍闪”不应继续归因到 mask path，要查弹幕 runtime。

11. 2026-06-08 遮罩反向显示的纠正

本节结论：最新日志已经证明数据链路是通的：Webmask segment cached 连续出现，Host 也持续输出 mask state=masked masked=true。随后实机反馈证明 DST_OUT 会反向显示：人物区域有弹幕，其他区域无弹幕。因此最终合成方向应回到 DST_IN，只保留解析后的可显示 path。

12. 2026-06-08 继续深挖：Chronos 包与 mask 消费边界

当前状态：不能说“参考整个智能防挡链路已经 100% 清楚”。已经闭环的是主进程 RPC、开关方向、clock 转发、native shader 合成语义；还没有闭环的是动态 DFM Chronos 包内部如何下载/解析 maskUrl，以及它在哪个 JS/native 节点把结果绑定为 cron_mask_texture。

12.1 新增已证实点

证据位置结论
ChronosPackageManager.Service.DFM smali_classes8\com\bilibili\common\chronoscommon\ChronosPackageManager$Service.smali DFM 的服务 key 是 service_danmaku。弹幕 Chronos 包不是随便一个通用包，而是按服务 key 获取。
suspendLoadPackage(pkgUrl, md5, sign, service) ChronosPackageManager.java 如果外部传入 pkgUrl，优先下载该包；否则通过 MOSS 拉取服务包信息，再下载、校验、缓存。
runPackage(pkg, null, argsMap) EnhancedChronosFragment$tryRunPackage$1.smali 包运行时会把 Fragment arguments 转成 Map 传入 Chronos。它不是单纯运行 APK 内置脚本。
URLRequest + 文件 parser UrlRequest$Request.java、plugins\C21301j.java Chronos 包可以请求远程 URL，文件响应会写到 download_files/chronos_file_x。这解释了为什么静态 APK 包里找不到完整 webmask 消费代码。
UnzipFile + AES 解密 plugins\P.java Chronos 包可以解压或解密资源包。mask/模型/脚本资源可能由动态包运行时拉取。

12.2 内置 cron_package 的负证据

E:\MyTool\MyBLBL\.trae\bili\out\.analysis\cron_package 是 APK 中提取出的 danmaku-flame-master-mini，info.json 显示版本 3.0.0，commit cdf962a9ce3201c8ce027a8c9ae840e61a70678d。

搜索项结果解释
VideoMask / maskUrl / mask_url只在 proto 定义处出现。说明 mini 包知道字段结构，但没有在可见代码里消费它。
URLRequest / UnzipFile框架能力存在，但未见与 maskUrl 绑定。不能把 mini 包当完整参考防挡实现。
GetDmView reply未见对 dataMap reply 的完整业务消费。真实消费逻辑更可能在动态 DFM 包，或在 native/插件层。

12.3 目前严格对齐时不能再猜的点

  • block_mask 不能按字面理解。参考是 block_mask = !getDanmakuAvoidFigure()，开智能防挡时发给 Chronos 的 block_mask 反而是 false。
  • mask_enabled 已被标注废弃，严格对齐时应以 DanmakuFilterChange.Request.blockMask 和 GetDanmakuFilter.Response.blockMask 为主。
  • 内置 MiniDmPkg 不能作为完整防挡包复刻依据。它没有显式下载/解析 maskUrl 的证据。
  • DST_OUT 已被实机证伪；当前解析后的 path 在 MyBLBL View 合成里应按可显示区域处理，继续使用 DST_IN。
  • 官方最终不是父 View 裁剪，而是 cron_output = CRONDefaultShading() * mask_sample 的 alpha 乘法。MyBLBL 的 saveLayer + DST_IN 只是 View 体系下的近似。

12.4 还缺的最后一段证据

要真正说“完全清楚”，还需要拿到或反推出动态 service_danmaku Chronos 包内部实现，也就是：

  1. 它如何从 GetDmView 的 reply 附件解出 DmViewReply.mask.maskUrl。
  2. 它是否通过 URLRequest 拉取 .webmask，是否通过 UnzipFile 处理资源。
  3. 它如何处理缺帧、seek、buffering、segment 预取和旧帧兜底。
  4. 它在哪个 Chronos/Poros 节点把 mask 帧转成 alpha texture，并绑定到 cron_mask_texture。

因此，下一轮如果继续逆向，应优先找运行时缓存目录里的 chronos_pkg/service_danmaku 或 MOSS 返回的 DFM 动态包，而不是继续全局扫 APK。APK 静态层已经证明主进程只是 RPC 和包管理桥，真正的防挡消费逻辑不在当前可见 mini 包里。
